# Continuous Integration (CI) Tests in Sylva-Core

In the Sylva-Core CI pipeline, we perform various types of tests, including linters, security checks, and compliance validations. This document specifically focuses on the tests executed during deployments via GitLab CI.

**Note:**

- Static lint tests are not covered in this document.
- Unit tests are also excluded from this discussion.

## Inclusion of Tests in Sylva-Core CI

Currently, any test jobs defined within the `.gitlab/ci/tests` directory are included in all deployments, unless explicitly disabled using the `SKIP_TESTS: "true"` variable (for example, in CAPD deployments).

The `tests` directory is organized into multiple subfolders to group tests related to specific areas, such as security and login. Additionally, a special `base` folder contains reusable definitions and a unique test that provides essential files (e.g., kubeconfig files) to other tests.

## Adding New Tests to Sylva-Core CI

When a developer wishes to create a new test, they should follow these steps:

1. **Determine Folder Structure:** Decide whether the new test can be placed in an existing subfolder or if a new subfolder is needed for clarity.
2. **Create a New YAML File:** Create a new `.yml` file for the test, ensuring the filename is descriptive.
3. **Define Test Job Names:** Assign names to the test jobs following the naming conventions outlined below.

### Naming Conventions

- Use only lowercase letters, with `-` as a separator.
- Prefix the job name with either `mgmt-` (for management cluster tests) or `wkld-` (for workload cluster tests).

For tests that need to be executed on both management and workload clusters, it is advisable to consolidate common elements using a hidden job definition (starting with a `.`).

Each job should utilize the `extends` keyword to inherit from either `.test-base-management-cluster` or `.test-base-workload-cluster`. These are defined in `.gitlab/ci/tests/base/base.yml`, which automatically:

- Sets the test stage to `deployment-tests`.
- Configures the appropriate tags for GitLab runners.
- Establishes rules for executing the test based on the presence of the corresponding cluster.
- Prevents the download of artifacts from previous stages, saving CI time.
- Adds a dependency on the `prepare-tests-env` job, which provides the `management-cluster-kubeconfig` and `workload-cluster-kubeconfig` files if available.
- Sets the `KUBECONFIG` variable to the appropriate kubeconfig file based on the job extension used.

### Example

Here is an example of how to structure a new test YAML file:

```yaml
.my-new-test-common:
  script:
    - my command $CLUSTER

mgmt-my-new-test:
  extends:
    - .my-new-test-common
    - .test-base-management-cluster
  variables:
    CLUSTER: management
    DESCRIPTION: Checking if command works.

wkld-my-new-test:
  extends:
    - .my-new-test-common
    - .test-base-workload-cluster
  variables:
    CLUSTER: workload
    DESCRIPTION: Checking if command works.
```

In this example, the `wkld-my-new-test` job will only be created if the workload cluster is deployed. The `KUBECONFIG` variable will be set to use the `workload-cluster-kubeconfig` file, which is automatically retrieved from the `prepare-tests-env` job.

It's also required to add `DESCRIPTION` variable for each tests to describe their purpose. This will be used to update [Tests description](#tests-description).

### Customizing Default Behavior

In rare cases, you may need to adjust the default configuration. For instance, if your test requires parsing information from the deployment dump, you will need to override the default behavior accordingly.

By following these guidelines, developers can effectively contribute to the Sylva-Core CI testing framework, ensuring robust and reliable deployments.

## Tests description

### Tests run on management cluster

| Name | Description |
|------|-------------|
| `mgmt-image-ref-report`|Generate a list of all images used during the deployment and upload it as an artifact.|
| `mgmt-GUIs-test`|Test basic authentication login against the GUI.|
| `mgmt-sso`|Test Single Sign-On login against the GUI.|
| `mgmt-login`|Run Hurl Login Tests.|
| `mgmt-thanos-alert`|Check if any critical alerts were raised by Thanos during deployments|
| `mgmt-cis-scan`|Retrieve the CIS scan report generated by the CIS Operator for RKE2-based deployments.|
| `mgmt-firewall-verify`|Check for packets received by the management cluster which didn't match any network policy|
| `mgmt-kube-score`|Execute the Kubescore tool on the cluster dump generated at the end of the deployment.|
| `mgmt-policy-report`|Verify if any PolicyReport is in a FAIL, WARN, or ERROR state.|
| `mgmt-cluster-policy-report`|Verify if any ClusterPolicyReport is in a FAIL, WARN, or ERROR state.|
| `mgmt-leak-report`|Inspect pod logs for any leaks of Kubernetes and Vault secrets.|
| `mgmt-cert-manager`|Test certmanager unit.|
| `mgmt-test-units`|Assess the status of test units after the management cluster deployment.|
| `mgmt-update-test-units`|Assess the status of test units after the management cluster update.|
| `mgmt-cronjob-testing`|Confirm that each Kubernetes CronJob can be executed successfully.|
| `mgmt-drift-corrections`|Check for drifts in the helm controller logs in the mgmt cluster|
| `mgmt-avoid-mgmt-deletion-policies`|Ensure we can't delete key Sylva Flux and CAPI resources for the mgmt cluster.|
| `mgmt-verify-longhorn-cleanup-events`|Highlight longhorn-instance-manager-cleanup events.|
